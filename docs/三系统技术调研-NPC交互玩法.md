# NPC / 交互 / 玩法机制 三系统技术调研（2026-06-12 · 三路研究编队，全带来源）

> 服务于：业务流程图 → 产品架构图 → PRD。每条结论标注【已验证】/【推断】。

## 总论：用户问的"编辑器最终目的=搭出完整游戏"对不对？

**对，且品类已经替我们裁剪好了范围。** 我们做的品类叫"叙事探索"（walking sim/叙事冒险）：机制极简（移动+走近交互）、反馈闭环完整。品类公认 core loop：**探索 → 发现（线索/对话）→ 记录外显（日志可见增长）→ 理解/解锁（新区域显影）→ 再探索**。07 号 demo 的"线索→倒计时→显影"已是这个 loop 的骨架，缺的主要是"记录外显"一环（调查日志）。战斗/数值成长是品类反模式（To the Moon 在 RPG Maker 上主动砍光战斗）——**不做不是偷懒，是品类正确**。

## 一、NPC 系统（六个子系统 + 分级）

行业完整构成：①对话 ②日程/作息 ③行为模式 ④好感度/关系 ⑤记忆/状态持久化 ⑥NPC 间互动。
我们已有 ①+⑤ 的简化版。让世界"活"起来靠 ②③④，全是**数据驱动薄层，不是 AI**。

**必做三件：**
1. **行为枚举**（抄 GB Studio 的 4 态下拉）：`static | face | wander | patrol`（patrol 带 waypoints）——引擎复用现有 grid 移动，编辑器 Inspector 一个下拉框
2. **条件对话页**（抄 RPG Maker 事件页）：`pages:[{when:{talked,cluesAtLeast},dialogue}]`，治"说一遍后台词永远不变"；条件源=现有 localStorage，零新状态
3. **AI 角色卡提取**（创作时一次性，mock 起步）：上传小说→抽角色/人设/关系/候选台词→人工确认。注意 chargraph 的 HN 实测教训：LLM 会编造关系，必须"AI 初稿+人工确认"

**后置**：日程系统（降级为"按章节进度换站位"）、inkjs（对话树>20 节点再换；【已验证】inkjs v2.4.0 UMD 243KB 无构建可接，MIT，2026 仍活跃，有 Phaser 官方集成案例）、好感度。
**永不做**：行为树/GOAP、运行时 AI 对话（花钱红线+毁调性，Inworld/Convai 调研确认全是按用量计费）、NPC 间真模拟。

## 二、交互系统（触发器 × 条件 × 指令集）

**行业地基**（RPG Maker 教科书机制，已验证官方文档）：事件页+条件分页 = 叙事状态机的视图层；开关/变量/自身开关三件套 = 状态地基。我们的"线索+requires:N"已是简化版 switch+variable——**地基踩对了，缺的是通用化并暴露给编辑器**。

**触发器 4 种**（覆盖用户要的区域/建筑/NPC/物体四类交互）：
| 触发器 | 现状 → 升级 |
|---|---|
| `interact` 走近按空格 | NPC/物件已有 → 统一为"挂事件" |
| `enter` 踏入区域 | 大门已有（单格）→ 扩成矩形区域可挂任意事件 |
| `leave` 离开区域 | 新增（"离开时 SE-12 喊住你"，叙事价值高） |
| `auto` 进图自动 | 新增，默认 once（开场过场/初见旁白） |

**条件 3 种 + once**：`flag`（含线索）/ `count`（兼容 requires:N）/ `visited`；`once`=触发过不再触发。**不做**完整事件页 UI——"事件数组+第一个条件命中的执行"对小说作者更好懂。

**指令 8 条起步**：`text / choice / set / if / teleport / fx / wait / end`（move/emote/showHide 入 v2）。
**实现范式**【已验证】：仿 RPG Maker `Game_Interpreter`（指针+step()+waitMode），~100 行；现有 eventDemo 播放器就是微型解释器，扩指令+挂触发器即可。**没有现成轮子**（JS 叙事事件解释器无事实标准库，RPG-JS 带全套构建链 overkill）。

**编辑器形态**：触发区=画板半透明矩形（推广 .ed-gate 范式）；事件脚本=现有事件编辑器改造成 **GB Studio 式指令积木列**（嵌套缩进可折叠，"➕添加一步"分类弹单）。把"事件主导模式"和"地图交互"统一成一个事件系统——平台叙事从"两个功能"变"一个引擎两种入口"。

**数据模型**：统一 `events` 字段（NPC/物件/章节均可挂），dialogue/interactText/gates 全不动视为语法糖。详细草案见调研原文（events: on/zone/once/when/run 结构）。

## 三、玩法机制系统（T0-T3 分级）

**状态管理三板斧**（ink 官方文档提炼，已验证）：①少量全局事实变量（非路径记录）②"到过/读过"自动计数当免费状态 ③资源必须叙事原生（80 Days："剧情免费送了金钱和行李"——我们的线索/倒计时同理）。

| 级 | 内容 | 要点 |
|---|---|---|
| **T0 已有→产品化** | 线索/倒计时/解锁/分支痕迹/存档 | 线索升级为有脸的"档案卡"（title/desc/icon）+ 编辑器"线索定义表"；倒计时档位入 settings 可配 |
| **T1 标配必补** | ①**调查日志**（对标 Outer Wilds Ship Log：当前目标角标+日志面板，`objectives:[{text,doneWhen}]`）②**叙事道具**（不做背包管理：`givesItem`/`requiresItem`="钥匙开门"闭环）③**章节推进仪式感**（目标完成提示，纯表现层） |
| **T2 可选留扩展位** | 好感度（与线索同构）/ 收集成就 / 多结局（用现有 branches 条件汇聚，零新系统） |
| **T3 明确不做** | 战斗/数值成长/商店（品类反模式）、实时昼夜（工程黑洞） |

**平台差异化总结**【推断但有据】：橙光给作者的是"变量/数值操作"（被教程吐槽"靠近编程思维"），凤翎工坊给的应该是**叙事原生命名**——"线索/调查目标/档案/解锁条件"，每个都是小说作者已有的心智概念。Bitsy 纪律：暴露概念≤5 个。
