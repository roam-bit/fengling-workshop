## 共同硬规则（每张都要）

1. **背景一律纯绿幕**：背景必须是**一整片纯绿色**（提示词已写好 solid pure green #00FF00）。你的工具出不了透明 PNG，绿幕是这个项目一直在用的老办法（我负责抠绿）。**别要求透明底，别用棋盘格底**。
2. **工具里选比例，分辨率不用管**：每张卡写了"工具里选什么比例"。生成出来是 1024 还是 2048 都行，**最终尺寸我来缩**（全部是 16 的倍数，已逐一核过）。
3. **30 秒像素粗筛**：出图后放大到 400% 看边缘——合格 = 一格一格的硬台阶；不合格 = 模糊渐变、毛玻璃边。**拿不准就直接给我，我跑行程统计脚本验**，不合格我会指出重 roll 哪张。
4. **物件统一三查**：① 背景是一整片纯绿、绿色没混进物体本体 ② 画面里只有这一个物件——没有地面、没有影子、没有角色 ③ 400% 看边缘是硬台阶。
5. 整体冷蓝灰调；**全批次禁琥珀禁暖棕**（琥珀是凤翎专属的全画面唯一暖色，物件绝不能碰）；病态黄绿也禁用（这两个房间没有异常侵染）。屏幕 / 指示灯一律青绿 #6fd3c4。
6. **比例锚**：本批全是室内家具/设备，没有建筑大门，批次 1 的"门高 24-32px"锚不适用（"门所在宽度选奇数格"同理不适用）。改用桌面高度感：这些物件给 48px 高的调查员用——**终端屏约半人高、桌面约腰高**，提示词里已带 waist-high / chest-high 锚。

## 落位契约（已用 python 对 chapters.json 实测——pos 一格不动，对话/线索引用着这些 id）

> tile 模式房间的碰撞 = 四周边界墙 + `innerWalls`（buildRoom 烧入），**不是** walkable 字符串。下表"碰撞实况"= innerWalls 对物件占格的实际覆盖（脚本逐格核对）。
>
> **⚠️ 时序守卫（P1 · 集成时必查，不查会假阴性）**：本表成立的前提是「ch1/ch2/ch3 的 room 里今天没有 `walkable` 键」（python 实查 2026-06-11 ✓）。但编辑器 `?edit=` 一保存就会把整格网序列化成 `walkable` 发给 server（index.html saveChapter L1446-1463），server.py L103 无条件写入；而 buildRoom（index.html L1991）**walkable 优先、有它就完全无视 innerWalls**。一旦集成前后有人对这三章跑过一次 ?edit= 保存，本表的 innerWalls 补丁就是死数据、5 处穿模洞复活，而只读 innerWalls 的 BFS 脚本会照样报绿（本项目踩过的「验证绿但游戏没变」同款）。所以：
> 1. **集成补丁脚本开头加守卫**：对 ch1/ch2/ch3 逐房 `assert 'walkable' not in room`；若已存在 walkable，改 walkable 字符串里对应格，**而不是**改 innerWalls。
> 2. **BFS 验证脚本必须复刻 buildRoom 优先级**：有 walkable 读 walkable，否则边界墙+innerWalls——与引擎读数路径完全一致，验证才作数。
